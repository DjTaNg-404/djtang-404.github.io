## 产品概述

阿城是一款突破次元壁的沉浸式桌宠 AI 学习搭子。它不是单纯给答案的问答框，而是围绕一场学习对话展开完整闭环：先识别用户意图，再决定是直接讲解、理解评估、追问引导、计划生成，还是阶段总结。产品的核心理念是“一场对话，即是一次学习”，目标是把碎片化学习重新组织成可持续推进的学习过程。

### 阿城创意来源：重塑深度学习的全场景陪伴体验

针对当前通用大模型在教育场景中存在的交互割裂、引导缺失、情感冷漠、记忆碎片化四类问题，阿城在产品层面做了四个方向的重构：

#### 交互破局：从切换窗口到触手可及

- 痛点：传统学习需要在资料、笔记与 AI 对话框之间频繁切换，打断心流并增加认知负荷。
- 方案：阿城以虚拟桌宠形态常驻屏幕角落，无论用户阅读文档还是编写代码，都能直接唤起问答，不需要跳转页面或复制粘贴。

#### 认知重塑：从直接给答案到苏格拉底式引路

- 痛点：通用模型倾向直接输出结果，用户容易产生“看懂了”的错觉，却没有真正完成推导和内化。
- 方案：阿城通过追问、类比、反向质疑和逻辑拆解，把浅层接收转化成主动思考。

#### 情感陪伴：从冷冰冰的工具到有温度的搭子

- 痛点：长期独自学习容易疲劳和分心，传统聊天式 AI 缺乏状态感知与陪伴感。
- 方案：阿城以拟人化桌宠提供静默陪伴、提醒休息、鼓励打气，并通过不同姿态与动画反馈学习状态。

#### 专属记忆：从会话即焚到时间线知识图谱

- 痛点：传统聊天记录零散，聊完就散，很难回溯知识脉络。
- 方案：阿城自动提炼对话中的高光时刻与核心结论，按时间线生成结构化学习笔记，并继续沉淀为知识图谱与长期学习资产。

## 产品亮点

### 无处不在的陪伴：沉浸式多模态桌面物理交互

传统大模型的交互经常需要在 IDE、阅读器和浏览器之间切换，直接打断当前工作流。阿城通过桌宠形态把问答入口放回桌面，让交互真正嵌入学习现场。

- 无需切屏的全桌面拓扑交互：基于 PyQt6 的桌宠可以识别并停留在当前 PDF 或 IDE 窗口边缘，以极低资源占用的方式完成后台陪伴。
- 即看即问的多模态自然输入：用户双击桌宠即可弹出支持 Markdown 与 KaTeX 渲染的透明对话框，点击可触发语音输入，实现边看边问。

### 陪你思考：基于 LangGraph 的多角色启发式伴读

阿城并不是一个固定话术的单角色聊天机器人，而是由多个学习角色协同完成整条答疑链路。

#### a) 启发式探究：授人以渔的苏格拉底导师

| 交互对象 | 回答策略与内容表现 |
| --- | --- |
| 用户提问 | 为什么我的模型训练损失一直在下降，但验证集损失却开始上升了？是哪里算错了吗？ |
| 传统 AI | 直接给出结论：这是过拟合现象，并立刻展开概念解释。 |
| 阿城 Inquiry | 先通过生活化类比和反问，引导用户主动判断模型是否只是记住了训练集，而没有真正学会泛化。 |

#### b) 费曼技巧校验：严格多维的学术裁判

| 交互环节 | 内容表现 |
| --- | --- |
| 用户复述 | 我理解的注意力机制，就是给句子里的不同单词分配不同的权重。 |
| 阿城 Judge | 从准确性、逻辑性、完备性三个维度给出反馈，并追问 Query、Key、Value 在权重计算中的角色。 |

#### c) 专属知识解答：扎实可靠的随身智库

| 能力定位 | 内容表现 |
| --- | --- |
| Tutor 解答 | 在明确需要知识输入时，直接输出结构化定义、示例与解释。 |
| 技术特征 | 依托上下文记忆与历史沉淀，输出连续、针对性强、幻觉更低。 |
| 交互目标 | 帮助用户快速建立概念框架，降低理解成本。 |

### 陪你坚持：领域知识图谱与动态抽取引擎

为了让学习成果被真正看见，阿城在后台持续做两件事：

1. 实时实体识别：在自然交流中静默抽取关键学术实体、代码模式和数学公式。
2. 知识关联编织与 RAG 增强：把新概念与已有知识网络连起来，既增强可视化沉淀，也增强后续检索与上下文召回。

### 陪你规划：可视化全景学习大盘

阿城把目标规划与大模型伴读放在了同一条链路里：

1. 智能任务树：系统会从问答中提炼并生成任务，将宏大的学习目标拆解为可执行的小步计划。
2. 时间线总结：系统会把阶段性学习轨迹以结构化时间线呈现，帮助用户回顾认知盲区、掌握进度和下一步重点。

## 技术架构概述

### 总体架构分层

本系统采用端云结合与多端分离的分层架构，将重度交互的前端体验与高耗能的云端认知引擎解耦，从而兼顾桌面端低资源占用和整体系统扩展性。

- 表现与交互层：Web 端基于 React + Vite，桌宠端基于 PyQt6。
- 接口与服务入口层：通过 FastAPI 统一承接多端请求并路由至底层能力模块。
- 核心认知层：基于 LangGraph 的多角色 Agent 工作流。
- 数据与记忆底座：以本地 JSON、Markdown 和 ChromaDB 向量检索库为主。

<Figure src="/img/products/acheng/总体架构分层.png" alt="总体架构分层" caption="双端交互、后端认知引擎与数据底座之间的分层关系。" />

### 核心模块组成

| 模块层级 | 核心组成 | 主要职责 | 对应价值 |
| --- | --- | --- | --- |
| 表现与交互模块 | Web Dashboard、桌宠客户端 | 提供任务管理、学习对话、历史查看与笔记回顾等交互入口 | 支持多终端接入，提升灵活性与陪伴感 |
| 接口与服务入口模块 | 对话服务接口、学习计划接口、笔记与历史接口 | 统一接收请求并路由到后端能力模块 | 解耦前端交互与后端逻辑 |
| 核心认知模块 | Analyzer、Tutor、Judge、Inquiry、Plan、Summary | 完成意图识别、知识讲解、理解评估、计划生成与总结输出 | 支撑连续式学习陪伴 |
| 数据与记忆模块 | 会话记录、任务与计划、学习笔记、学习画像、向量检索库 | 保存关键状态和历史数据，支持个性化与长期沉淀 | 实现长期学习追踪 |

### 系统主流程与数据流转

阿城围绕“请求接入 -> 状态恢复 -> 意图路由 -> 认知执行 -> 结果生成 -> 学习沉淀”这条主链路运行，而不是只做一次输入和一次回复。

| 阶段 | 系统动作 | 输出结果 |
| --- | --- | --- |
| 请求接入 | 接收来自 Web Dashboard 或桌宠客户端的用户输入，并识别当前任务与会话上下文 | 形成当前请求的基础状态 |
| 状态恢复 | 从本地存储中读取历史消息、上下文摘要、任务状态与相关学习数据 | 重建本轮对话所需的学习上下文 |
| 意图路由 | 由 Analyzer 判断属于问答、评估、追问、总结还是计划生成 | 确定后续执行链路 |
| 认知执行 | Tutor、Judge、Inquiry、Plan 等模块按需激活 | 生成多个认知子结果 |
| 结果生成 | Aggregator 组织各模块输出，必要时触发 Summary | 返回面向用户的最终响应 |
| 学习沉淀 | 更新会话记录、摘要、画像、任务计划、笔记与索引 | 完成学习状态更新与长期数据沉淀 |

## 核心功能技术

### Agent 架构编排

阿城没有采用单模型单提示词的问答模式，而是以 Analyzer 作为意图路由中枢，再把 Tutor、Judge、Inquiry、Plan、Summary 串成可控的认知工作流，最后由 Aggregator 汇总结果。

<Figure src="/img/products/acheng/Agent架构编排.png" alt="Agent 架构编排" caption="多角色节点分工协作，保证学习支持方式可控且可扩展。" />

| 节点 | 主要职责 | 在学习场景中的作用 |
| --- | --- | --- |
| Analyzer | 对用户输入进行意图识别，并决定进入哪条处理链路 | 系统的任务路由中枢 |
| Tutor | 对知识点进行讲解、解释与结构化回答 | 教学输出核心节点 |
| Judge | 对用户理解、复述或回答进行评估与诊断 | 发现认知漏洞和偏差 |
| Inquiry | 基于当前对话状态提出进一步问题 | 通过追问推动深入理解 |
| Plan | 根据目标、任务状态和上下文生成或调整学习计划 | 支持目标驱动学习 |
| Summary | 在总结场景下生成总结与笔记内容 | 沉淀阶段性学习成果 |
| Aggregator | 对多个认知模块输出进行整合 | 保证最终输出连贯一致 |

### 对话上下文与记忆机制

系统不会把完整历史对话一次性送入模型，而是采用三层上下文组织机制：

- 短期工作记忆：保留最近若干轮原始消息，维持当前对话连贯性。
- 长期认知链：在阈值触发后自动摘要压缩旧消息，保留长期学习脉络。
- 关联历史召回：基于向量检索优先、轻量相似度匹配兜底，从历史中召回最相关片段。

<Figure src="/img/products/acheng/对话上下文与记忆机制.png" alt="对话上下文与记忆机制" caption="窗口记忆、摘要压缩与历史召回共同组成长期学习上下文。" />

| 记忆层级 | 实现方式 | 主要作用 | 当前实现特点 |
| --- | --- | --- | --- |
| 短期工作记忆 | 保留最近若干轮原始消息 | 维持当前对话的连贯性，支持指代消解和即时理解 | 默认保留最近 12 条消息 |
| 长期认知链 | 在阈值触发后对旧对话进行摘要压缩 | 记录学习过程中的核心认知脉络 | 达到阈值后自动更新摘要，并保留最近 5 条消息不参与压缩 |
| 关联历史召回 | 基于向量检索优先、相似度匹配兜底 | 补充与当前问题相关的历史背景信息 | 优先使用语义检索，必要时回退到 Jaccard 相似度 |
| 持续状态更新 | 会话写回、本地存储、向量索引更新 | 将每轮对话转化为后续学习可用的长期状态 | 每轮对话结束后自动保存并更新索引 |

### 学习计划与计划生成机制

系统以 task_id 和 session_id 作为任务与会话的组织单元。用户既可以在 Web 端显式创建学习任务，也可以在自然对话里表达学习目标、时间安排和阶段要求，由 Analyzer 将请求路由到计划生成链路。

### 会话历史、笔记与学习沉淀机制

阿城将对话视为最小学习单元，以天为阶段性归档单位，以任务为长期目标单位。原始会话用于回看，阶段总结用于沉淀笔记，任务级笔记用于回顾长期学习进展。

### 学习画像与知识扩展机制

系统会在每轮对话结束后抽取能够反映学习目标、偏好、薄弱点和阶段进展的信息，并将画像摘要重新注入后续提示词；同时可基于完整会话构建知识图谱，进一步增强可视化与长期分析能力。

### 数据存储与管理机制

当前阶段系统采用以本地文件为核心的轻量数据存储方案：

- 会话数据：结构化 JSON 文件
- 任务计划与任务索引：长期目标与组织关系
- 学习笔记与总结：Markdown 或结构化任务数据
- 学习画像：独立的结构化卡片数据
- 向量检索索引：按任务维度维护历史召回能力

## 多终端交互形态

### Web 端技术实现

Web 端基于 React 与 Vite 构建，是当前最完整的学习工作台。用户可以围绕同一任务创建计划、持续对话、回看历史、整理每日笔记与任务笔记，形成连续学习链路。

### 桌宠端技术实现

桌宠端基于 PyQt6 构建，通过悬浮窗、输入框、动态状态切换和桌面行为反馈，把阿城从浏览器工具扩展成常驻桌面的学习伙伴。桌宠只负责输入采集与状态展示，真正的意图分析、认知执行和学习状态更新仍由统一后端完成。

### 多端统一后端架构的价值

阿城并不是在 Web 与桌宠上分别维护两套能力，而是围绕同一套智能后端提供不同入口。这样做的直接收益是：

- 产品层面：Web 端适合深度学习与长期管理，桌宠端适合高频陪伴与即时互动，两者互补。
- 工程层面：统一后端减少重复建设，降低维护成本，并为未来新增交互入口保留扩展空间。
