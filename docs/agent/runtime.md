# 我如何实现一个轻量但可控的 Agent Runtime

## 0. 背景

这段时间 OpenClaw 小龙虾系列的产品爆火，身边无论是技术同学还是非技术的同学都有在养小龙虾。我在本地试用了 CoPaw。但简单使用之后，我便没有兴趣了：一是我的终端必须一直在后台运行；二是我多次尝试与 iMessage 适配均失败，造成很大的挫败感；最关键的是，我并没有真正能用得上它的场景。所以我觉得大动干戈去本地部署这么大的一个应用，似乎投入和收获并不成正比。

前段时间开发了阿城，也就是 ChatTutor，一个面向对话学习场景的桌宠，并带着阿城参加了 AI Hackathon Tour 后，我就在想，能不能让桌宠的内核变成 Claw 系列的产品？但如果还是走部署 OpenClaw，这样还是需要占用很大的磁盘空间和资源。那我能不能做一个轻量化的 Agent，作为我们桌宠的核心呢？恰逢 Datawhale 开源了 Hello-Claw 课程，在简单过了一遍课程后，做一个轻量化 Agent 内核的想法就出来了。

现在我的 AI 桌宠项目 Desktop-Claw 已经简单能够跑了，同时也借这个机会来沉淀一下具体的技术。

## 1. 为什么 AI 应用需要 Runtime？

一次 LLM 调用指的是通过一次与 LLM 交互来获得结果，适合完成单轮、单步问题。这时第一个问题就出现了，每一次调用之间都是独立的，状态不会自动延续。而想要让多次调用形成连续任务，只能靠外部上下文来补充。第二个问题是，当我们希望 LLM 能够自主调用工具解决任务时，单次调用并不能完整覆盖工具调用场景。因为在 LLM 使用工具的时候，tool call 发生后，执行状态如何切换？模型调用是否结束？Tool result 返回后应该如何继续后续推理？第三个问题是，当一次调用 LLM 包含多个任务时，是一次调用直接把所有任务都完成，还是多次调用 LLM？

当我们的系统需要从单轮问答变成多步执行、模型开始需要调用工具和处理中间状态时，便需要引入 Agent Runtime。因此，Runtime 便是让 Agent 能够真正跑起来的执行系统。

正因要解决如上状态管控、执行控制、调用粒度的问题，Runtime 需要承担这些职责：

- 接收任务
- 组装上下文
- 调用模型
- 解析 tool call
- 执行工具
- 把结果喂给模型
- 控制循环终止、超时、取消、错误处理

Runtime 解决的并不是模型能不能回答，而是任务能不能稳定执行。

## 2. 设计目标：轻量、可控、可拓展

许多场景下，任务需求并不需要复杂的 Agent 来解决。在这种场景下，重型通用超级 Agent 的许多能力可能一直都不会用上。而桌宠是一个高频、常驻、轻交互的场景，在这类场景下并不适合一个重型、复杂、资源占用高的 Agent 系统。若桌宠接入了通用的超级 Agent，则会出现能力溢出的状况，不必要的复杂能力进入日常场景，从而提高系统负担。

同时，由于桌宠是长期存在于用户桌面上，这导致它不能出现失控的状况，影响用户的正常使用。

再者，桌宠又不能完全写死，因为用户诉求会变化。用户对于桌宠会有不同的额外需求，不同用户也会对桌宠的陪伴性、任务能力有不同期待。

因此在这些约束下，我把桌宠 Runtime 的设计目标收敛成了三个词：轻量、可控、可拓展。

### 2.1 轻量

在 Runtime 设计里，“轻量”实际上并不是少做事，而是把系统能力收敛到场景真正需要的范围内。

具体而言，轻量即是：运行成本要低，不能干扰其余进程的日常运作；架构复杂度要低，运行轻巧且方便拓展；能力边界要收敛，以最低的能力范围完成大部分任务需求；有效解决高频、小任务、短链路的问题，不追求大而全的自动化，也不完成长链路任务。

轻量并不代表能力弱，而是主动收敛能力边界，以最低成本完成最基本的功能。因此在这个 Runtime 设计里，不需要做多 Agent 架构和复杂 planner。

### 2.2 可控

若桌宠不断地在思考，而没有跟你交互，则互动感会不明显；同时如果桌宠随意地修改、删除、转移你的文档，那可能会造成不可逆的状况；当桌宠需要完成多个任务时，没有合适的引导和管理，桌宠直接罢工等等，这些就是无界执行、工具权限越界、调度与任务管理失效的情况。

因此为了避免出现这些问题，在设计桌宠的 Agent Runtime 时就需要满足“可控”，即控制桌宠的执行步骤、工具调用边界和任务生命周期等。思考的最大步数要被明确限制，任务执行的 FIFO 队列要被纳入运行时控制，安全边界和工具使用能力要成为系统内建约束。

### 2.3 可拓展

“可拓展”指的是在初步设计 Runtime 时，并不需要把所有能力都做进去，而是架构上预留出演进空间。因为不同用户对于桌宠可能会有不同需求，有人需要桌宠更注重陪伴性，也有人更注重让桌宠的任务能力更强，因此在设计时需要让 Runtime 做到可拓展。

详细而言，Runtime 的设计要考虑到以下几个拓展场景：

- 场景可拓展：用户可能不满足于桌宠只有陪伴对话的功能，因此需要引入轻任务执行。
- 能力可拓展：用户对于轻任务执行需要引入一些新的 skill 或 tool，因此要预留出接口。
- Memory 可拓展：聊天数据增多会引发对复杂长期记忆机制的需求，因此 Memory 机制也要做到可拓展。
- 人格可拓展：有用户在不同工作环境和任务下，希望桌宠能以不同性格来陪伴，因此设计 Runtime 时需要允许这类上层能力的挂接。

## 3. 整体架构

### 3.1 整体架构图

```text
Desktop UI (悬浮球 + 对话面板 + 日历视图)
        │
        │  WebSocket + HTTP
        ▼
   Gateway (入口层)
        │
        ▼
Task Coordinator (FIFO 任务队列)
        │
        ▼
  Agent Loop (ReAct 循环)
        │
   ┌────┼────────┐
   ▼    ▼        ▼
 File  Memory   ...未来扩展
Skill  Skill
        │
        ▼
Prompt Assembler (5 层 System Prompt 组装)
  │  SOUL.md · USER.md · CONTEXT.md
  │
  ▼
Memory Service (按天 JSON 归档 + 每日内化)
```

### 3.2 请求的主链路

1. 用户在 Desktop UI 中发起输入，请求通过 WebSocket 或 HTTP 进入 Gateway。
2. Gateway 接收请求后，将任务交给 Task Coordinator。Task Coordinator 负责把请求纳入队列，并调度进入 Agent Loop。
3. Agent Loop 负责驱动模型推理，以及工具调用和结果回填。
4. 在执行过程中，Agent Loop 会通过 Skill System 获取具体能力。
5. 同时通过 Prompt Assembler 组装当前执行所需的系统上下文。
6. 如果涉及历史信息和长期记忆，则会读取或写入 Memory Service。
7. 当任务完成后，结果沿链路返回到 UI，并在需要时同步更新记忆归档。

### 3.3 核心模块概览

- `Gateway`：负责接入 UI 请求，并对外暴露 Runtime 的通信接口。
- `Task Coordinator`：负责队列调度，以及任务的生命周期管理。
- `Agent Loop`：负责模型调用、工具调用与循环执行。
- `Skill System`：负责将可执行能力组织成可拓展的工具层。
- `Prompt Assembler`：负责将人格、上下文与运行时信息组装成系统提示。
- `Memory Service`：负责对话归档、历史回顾与长期记忆内化。

## 4. Runtime 核心模块设计

在介绍了 Runtime 的总体框架后，这一章开始解释这些模块各自解决了什么问题，以及如何协同构成一个可运行的 Runtime。

### 4.1 Gateway

Gateway 是统一入口层，位于 Desktop UI 和 Agent 内核之间。UI 不应该直接和 Agent Loop 耦合，因此需要引入 Gateway 作为统一入口，进行 UI 请求的接收，并把运行时状态回传回去。Gateway 不直接做复杂业务决策，而是负责把信息安全、可靠、可追踪地送到该去的地方，再把结果送回来。

简单而言，Gateway 通过广播 ack、token、status、done、error 等状态，告知 Desktop UI 当前工作处于什么状态，同时也能将核心状态和任务交给 Task Coordinator。

在 Desktop UI 和 Gateway 之间就是靠 HTTPS 或 WebSocket 来传递信息。HTTPS 擅长标准 REST API，生态成熟；WebSocket 擅长双向通信，适合 token 流式输出、实时日志和多事件会话。两种协议可以相互配合，组成桥梁。

### 4.2 Task Coordinator

在我们的桌宠场景下，处理的任务类型往往不止对话一种：用户可能会连续发送多条对话请求，同时也可能会有别的功能请求。而 Agent Loop 只负责“一个任务怎么跑完”。如果所有请求都直接打进 Agent Loop，那么就会出现上下文冲突、状态混乱、流式结果串线的问题。因此需要一层专门做调度，把“任务执行”从“任务接入”里分离出来，所以才会有 Task Coordinator 这一模块。

Task Coordinator 实际上就是 Runtime 的调度层，它负责的不是推理，而是任务进入系统后的生命周期管理，将“请求接入”和“任务执行”解耦。

在具体设计中，使用了 FIFO 串行队列。由于桌宠场景下的任务并不会太复杂，且单人用户下并不需要支持高并发，因此使用 FIFO 串行单任务执行队列。具体队列维护过程中，通过维护 `queue` 和 `running`，来对待执行任务与当前执行任务进行管理，同时也处理超时、取消和状态回调等情况。

为什么以前做基于 Python 的项目没有考虑 Task Coordinator？因为使用的是 FastAPI 这种 HTTP 请求-响应模型。每个 HTTP 请求天然隔离，框架已经帮我们管理了并发，从来不需要自己写队列。

### 4.3 Agent Loop

单次调用无法承接多步执行，在任务执行过程中，模型调用和工具调用之间需要来回切换，即需要一个能够持续驱动“推理 -> 调工具 -> 再推理”的执行循环。所以在 Runtime 中，Agent Loop 就是这个执行循环模块。

与 Task Coordinator 相比，Task Coordinator 管的是任务调度，Agent Loop 管的则是任务执行。也就是说，在 Task Coordinator 的 FIFO 串行队列中，Agent Loop 执行的就是被打上 `running` 标记的任务。

在具体执行过程中，模型会先思考当前问题，如果需要外部能力，就发起 tool call，此时 Runtime 执行工具，获得工具结果后再返回给模型，模型基于新的信息继续推理，直到输出最终答案。这个过程借鉴了 ReAct 的核心思想，Agent Loop 的核心就是一个 ReAct-like 的执行过程。

但与 ReAct 相比，Agent Loop 的核心是打造一个可运行、可中断、可控制的循环。其真正有价值的地方，是设置了思考的最大步数、可以进行状态提示、有取消和错误回调功能、有 history trim 和 tool result 回填。Agent Loop 并不是一个无限循环，而是必须被运行时机制约束的循环过程。ReAct 只是思路，真正让它在产品里可用的是这些控制机制。

### 4.4 Skill System

给桌宠引入工具调用能力时，如果把所有工具都一次性以 function calling 的 JSON schema 暴露给模型，随着工具数量增加，会带来几个问题：

- prompt 和 tools 上下文膨胀
- 模型选择工具的负担增加
- 不同场景下真正需要的能力很少

对于轻量化桌宠而言，这种全量暴露方式太重了。

现在很多 Agent 系统已经不再只依赖裸的 function calling，更常见的是把能力组织成 Agent Skills。参考 OpenClaw 这类产品的思路，结合自己轻量化的目标，选择了 Skill System。

Agent Skills 并不等于一个工具列表，其最主要的核心是“渐进式披露”，即不将工具一次性全暴露出来，而是逐步展开。系统先只给模型一个“有哪些能力方向”的概述，当模型判断需要某类能力时，再激活对应 Skill。激活后，相关工具、文本和参考文档才真正进入当前执行上下文。渐进式披露的目的，不是让模型知道更多能力，而是只在需要时知道恰当的能力。

对于桌宠来说，日常高频交互并不需要一整套重型工具系统，Agent Skills 机制能让 Runtime 在大多时候保持轻量，同时也可拓展。例如 memory 这类能力，也可以作为 skill 按需启动，而不需要让桌宠在每一轮中都注入或回顾。

因此在桌宠 Runtime 中，只设计了两个 Skills：

- `file skill`：负责最基础的文件 read/write/edit 功能。
- `memory skill`：负责历史回顾和记忆检索。

这两类 Skills 覆盖了桌宠场景下最基础、最常用的外部能力。

### 4.5 Prompt Assembler

对于 Agent 来说，系统上下文不是单一 prompt，而是由多种来源共同组成。如果没有统一的组装层，信息注入会混乱，优先级也会失控。桌宠的行为也要依赖人格、用户画像、长期记忆和当前机能状态，因此 Prompt Assembler 就是 Runtime 的上下文组织层。

在这个 Runtime 中，系统上下文被拆成了几层：一层是固定 Base Prompt，用来提供运行时信息、回复规范和文件操作约束；一层是 `SOUL.md`，用于定义桌宠的人格核心；一层是 `USER.md`，用于记录用户画像；一层是 `CONTEXT.md`，用于承载动态认知；此外还有 Skills 相关的提示信息，以及仅在首次启动时启用的 `BOOTSTRAP.md`。这些信息来源承担的职责不同，因此不能混在一起管理。

Prompt Assembler 也不是一个静态模板，而是一个动态上下文组装层。它会根据当前状态决定哪些信息应该进入本轮 system prompt。例如，当系统仍处于首次引导阶段时，会额外注入 `BOOTSTRAP.md`；当 `USER.md` 或 `CONTEXT.md` 仍是空模板时，则不会注入；而 Skills 相关的提示内容也会随着技能激活状态变化而动态调整。为了避免每轮执行都重复读取磁盘文件，这一层还通过缓存机制减少了不必要的文件读取。

因此，Prompt Assembler 在 Runtime 中承担的并不是“写 prompt”的职责，而是“组织上下文”的职责。它让系统上下文能够以分层、可控、可拓展的方式进入模型：对轻量来说，它避免了无序注入和上下文冗余；对可控来说，它明确了不同信息的边界和优先级；对可拓展来说，它也为后续新增上下文层预留了空间。

## 5. 总结

这篇文章所沉淀的，是一套面向桌宠场景的轻量 Agent Runtime。与其说它是在 OpenClaw 之上套了一层桌宠外壳，不如说它是在桌宠这一具体场景下，重新收敛和组织了一套适合长期运行的 Agent 内核。这套 Runtime 的重点不在于能力尽可能多，而在于让桌宠能够稳定地接收任务、调用能力、组织上下文并完成执行，同时在设计上尽量做到轻量、可控和可拓展。

为了让这套 Runtime 真正适合桌宠场景，我有意识地收敛了能力边界，把重点放在任务调度、执行循环、技能机制和上下文组织这些运行时内核上，而没有一开始就追求多 Agent、复杂 Planner 或更重的前端体验，也没有过早投入过多精力去打磨视觉层。对我来说，将桌宠的能力尽量 Skills 化，是实现轻量化的一项关键选择。

未来如果继续演进，我希望它在保持轻量的前提下，进一步变得更安全、更稳定，也能支持更多贴合桌宠场景的能力，去承载更自然的陪伴体验。除了继续完善前端形象和交互方式之外，我也希望尝试引入语音输入能力，进一步探索桌宠与 Agent 在人与系统之间更自然的交互形式。
